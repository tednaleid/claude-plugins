# just-bootstrap: Claude Code Plugin for CI/Release/Homebrew Infrastructure

**Created**: 2026-04-01
**Confidence Score**: 96/100
**Status**: Draft

## Security Note

No secrets are stored in this repo or any generated artifacts. All secrets
(HOMEBREW_TAP_TOKEN, APPLE_CERTIFICATE, etc.) live in GitHub's encrypted
secrets and are referenced only by name in workflow YAML.

## Context

Ted maintains projects across Rust (sumpig), Zig (veer), Swift macOS (montty),
and TypeScript/Bun (limn). Each has CI/release infrastructure that was set up
ad-hoc, resulting in inconsistencies: veer uses lightweight tags while sumpig
uses annotated tags with Claude-generated release notes; veer has no homebrew
while sumpig does; limn has pre-commit hooks but sumpig doesn't; etc.

The goal is a reusable Claude Code skill that can be applied to any repo to
audit what exists and generate what's missing, normalized to the best patterns
found across all four repos.

## Problem Statement

No repeatable way to bring a repo up to a consistent CI/release/distribution
standard. Each new project requires manually copying and adapting workflows,
and existing projects drift apart as patterns improve in one but not others.

## Goals

1. Create a `/just-bootstrap` skill that auto-detects language/build system and
   audits infrastructure, presenting a checklist of what exists vs what's missing
2. Let the user pick which components to set up (audit-then-menu flow)
3. Generate correct, copy-paste-quality CI/release/homebrew artifacts for Rust,
   Zig, Swift (CLI + macOS app), and TypeScript/Bun
4. Package as a plugin in `tednaleid/claude-plugins` (multi-plugin repo)

## Success Criteria

- [ ] `/just-bootstrap` on veer correctly detects Zig CLI, shows audit, offers to add homebrew + upgrade bump/retag
- [ ] Generated `just bump` creates annotated tags with Claude-generated release notes for all languages
- [ ] Generated `just retag` preserves tag annotations for all languages
- [ ] Generated homebrew formula (CLI) and cask (macOS app) templates match quality of sumpig/montty
- [ ] Generated setup script creates tap repo and prints PAT instructions
- [ ] Generated CI workflow runs `just check` on appropriate platforms
- [ ] Generated release workflow builds cross-platform, creates GitHub release with `--notes-from-tag`, and updates homebrew tap

- [ ] Plugin installable via `/plugin marketplace add tednaleid/claude-plugins`

## Scope

### In Scope

- Justfile creation/normalization (check, test, lint, fmt, build, install, bump, retag, install-hooks)
- GitHub Actions CI workflow per language
- GitHub Actions release workflow (cross-compile for CLIs, DMG for macOS apps)
- Homebrew formula (CLI) and cask (macOS app) templates
- Homebrew tap setup script with PAT instructions
- Apple code signing in release workflows (conditional, same secret names as montty/limn)
- Pre-commit hook via `just install-hooks`
- CLAUDE.md additions for CI/testing conventions
- Auto-detection of language, build system, project type, tag prefix convention
- Audit-then-menu progressive disclosure
- Optional enhancement suggestions (coverage badges, binary size checks, benchmarks)

### Out of Scope

- Non-GitHub CI (Jenkins, GitLab, etc.)
- Non-Homebrew package managers (apt, chocolatey, snap)
- Deployment workflows (Pages, cloud deploys)
- Windows targets (lowest priority, can be added later)
- MCP servers in the plugin

### Future Considerations

- Go, Python, Java language support
- Windows cross-compilation targets
- Dependency update bots (like montty's bump-ghostty.yml)
- Nix/Flake packaging

### Skill Authoring Best Practices (from skill-creator)

These best practices from the official skill-creator guide shape how we write
the skill:

- **SKILL.md under 500 lines.** Heavy content (templates, YAML examples) goes
  in references/. SKILL.md contains the workflow logic and clear pointers to
  which reference to read for each step.
- **Description is the primary trigger.** The frontmatter `description` field
  determines whether Claude invokes the skill. It must be specific about when
  to use it and slightly "pushy" (e.g., "Use this whenever the user mentions
  CI, release automation, homebrew, justfile, bump, retag, pre-commit hooks,
  or wants to normalize build infrastructure").
- **Progressive disclosure** -- three levels:
  1. Metadata (name + description) -- always in context (~100 words)
  2. SKILL.md body -- loaded when skill triggers (<500 lines)
  3. Bundled references -- loaded on demand (unlimited size)
- **Explain WHY, not just WHAT** -- avoid heavy-handed MUSTs. Explain reasoning
  so the model can adapt to edge cases rather than following rote instructions.
- **Imperative form** for instructions.
- **Include examples** of expected output (the audit checklist format, etc.)
- **Domain organization by variant** -- references/ organized by concern, Claude
  reads only the relevant reference file for the detected language.
- **Description optimization** -- after initial creation, we can run the
  skill-creator's optimization loop to improve triggering accuracy.

### SKILL.md Workflow Design

The skill follows a 5-step flow: DETECT -> AUDIT -> CHECKLIST -> GENERATE -> VERIFY

**Step 1: Detection.** Read repo to determine:
- Language/build system: `Cargo.toml` (Rust), `build.zig.zon` (Zig),
  `project.yml`/`Package.swift` (Swift), `package.json` + `bun.lockb` (TS/Bun)
- Project type: CLI vs macOS app (determines formula vs cask, signing needs)
- Tag prefix: check `git tag -l` for existing convention (v-prefix vs bare)
- Version file locations: per-language (Cargo.toml, build.zig.zon + src/main.zig,
  Info.plist, package.json)
- GitHub owner/repo: from `git remote get-url origin`

**Step 2: Audit.** Check what infrastructure exists:
- Justfile present? Which standard recipes exist?
- `.github/workflows/ci.yml` present?
- `.github/workflows/release.yml` present? Does it update homebrew?
- `scripts/setup-homebrew-tap.sh` present?
- Bump recipe: exists? Uses annotated tags? Has Claude release notes?
- Retag recipe: exists? Preserves annotations?
- Pre-commit hook: `.git/hooks/pre-commit` exists? `install-hooks` recipe?
- CLAUDE.md: has CI/testing conventions section?

**Step 3: Present Checklist.** Use AskUserQuestion with multiSelect showing
what's present ([x]), missing ([ ]), or partial ([~]) for each concern.

**Step 4: Generate.** For each selected item, read the corresponding reference
file and generate language-appropriate output. Generation order respects
dependencies: Justfile -> CI -> release -> homebrew -> bump/retag -> pre-commit.

**Step 5: Verify.** Run `just check` to confirm nothing broke. Suggest pushing
a branch to test CI.

### Reference Files Organization

References are organized **by concern** (not by language), with language-specific
sections inside each. This matches the skill's concern-driven workflow.

Each reference contains complete, copy-paste-ready templates derived from the
actual working files in sumpig, veer, montty, and limn.

### Key Patterns to Normalize (from exploration)

**Bump recipe** -- best pattern is sumpig's:
- Extract current version from canonical source
- Compute new version (patch increment default, explicit version optional)
- Update all version files, commit
- Generate release notes via `claude -p` with fallback to `git log | sed`
- Create **annotated** tag with notes embedded
- Push commits + tags automatically
- veer currently uses lightweight tags and manual push -- needs upgrade

**Retag recipe** -- best pattern is sumpig's:
- Save existing annotation before deleting
- Delete GitHub release, remote tag, local tag
- Recreate annotated tag from saved notes
- Push
- veer currently uses `git tag -f` (force, no annotation) -- needs upgrade

**Release workflow** -- normalize to:
- Use `gh release create` with `--notes-from-tag` (not `softprops/action-gh-release`)
- Fetch annotated tag explicitly (`git fetch origin refs/tags/...`)
- Compute SHA-256 and update homebrew tap (conditional on HOMEBREW_TAP_TOKEN)
- veer currently uses `softprops/action-gh-release` with `generate_release_notes: true`

**CI workflow** -- normalize to:
- Use `just check` as the primary CI command (match montty/limn pattern)
- Run on appropriate platforms per language
- Rust: ubuntu + macos, cargo test + clippy + fmt-check
- Zig: ubuntu, zig build test + zig fmt --check
- Swift macOS: macos-26, xcodebuild + swiftlint
- TS/Bun: ubuntu, bun test + eslint + tsc

**Homebrew** -- two patterns:
- CLI (formula): multi-arch tar.gz binaries, `Formula/NAME.rb` in tap
- macOS app (cask): single DMG, `Casks/NAME.rb` in tap
- Both: setup script creates tap repo + seeds + prints PAT instructions
- Both: release workflow computes SHA-256 and pushes updated formula/cask

**Apple code signing** -- conditional pattern from montty/limn:
- Only runs if `APPLE_CERTIFICATE` secret is set
- Import p12 cert, create keychain, store notarization credentials
- Re-sign with "Developer ID Application" + hardened runtime
- Ad-hoc fallback when no cert
- DMG signing + notarization + stapling
- Secret names: APPLE_CERTIFICATE, APPLE_CERTIFICATE_PASSWORD, APPLE_ID,
  APPLE_TEAM_ID, APPLE_APP_SPECIFIC_PASSWORD

### Bump/Retag Convention

Both `just bump` and `just retag` always take a **bare version number** without
a `v` prefix. e.g., `just bump 1.2.3` and `just retag 1.2.3` (never
`just bump v1.2.3`). The recipe itself adds the tag prefix based on the
project's convention. This is normalized across all languages.

For bump with no argument, patch increment is the default: `just bump` bumps
`1.2.3` to `1.2.4`. Explicit version is also supported: `just bump 2.0.0`.

### Optional Enhancement Suggestions

After generation, the skill suggests (but does not generate) enhancements that
make sense for the detected project type:
- Coverage badges (for TypeScript/Bun projects with vitest)
- Binary size checks (for Zig/Rust CLIs with small binary goals)
- Benchmark CI runs (for projects with benchmark suites)
- Dependency update bots (for projects with submodules or lockfiles)
